# Installation

This chapter introduces how to install the following logging plugins: **Alauda Container Platform Log Essentials**, **Alauda Container Platform Log Collector**, **Alauda Container Platform Log Storage for ClickHouse**, and **Alauda Container Platform Log Storage for Elasticsearch**.

:::warning

1. The `global` cluster can query the log data stored on any workload cluster within the platform. Ensure that the `global` cluster can access port 11780 of the  workload cluster.

2. **Alauda Container Platform Log Storage for ClickHouse** depends on **ClickHouse Operator**. Before installing the plugin, ensure that ClickHouse Operator has been uploaded to the cluster.

:::

## Installation Planning

:::info
Some components of the Logging are resource-intensive. We recommend running them on infra nodes and setting nodeSelector and tolerations to ensure they run only on those nodes. If you are evaluating the product and have not provisioned infra nodes, you can remove these settings so the components run on all nodes.

For guidance on planning infra nodes, see <ExternalSiteLink name="acp" href="/configure/clusters/acp-node-planning.html" children="Cluster Node Planning" />.
:::

Alauda Container Platform Log Storage plugins can be installed in any cluster, and any cluster's log storage component can be selected for log collection to interface with the storage data.

So , before installing the log storage plugin , you need to plan the cluster and node where the log storage component will be installed.

  - Avoid deploying log storage plugins in the global cluster. Instead, deploy them in workload clusters to ensure management cluster failures do not disrupt log-based issue resolution.

  - Prioritize centralizing logs to a single log storage cluster. If log volume exceeds maximum capacity thresholds, distribute logs across multiple storage clusters.

  - Deploy at least one log storage instance per network zone to aggregate logs locally, minimizing cross-data-center public network traffic (which incurs high costs and latency).

  - Dedicate exclusive nodes for log storage, avoiding co-deployment with other applications or platform components. Log storage requires high I/O throughput and may be affected by interference.

  - Mount dedicated SSD disks for log storage to significantly enhance performance.


## Install Alauda Container Platform Log Essentials via console

1. Navigate to **Marketplace** > **Cluster Plugins** and select the `global` cluster.

2. Click the action button to the right of **Alauda Container Platform Log Essentials** and select **Install**.

3. Click **Install**.

## Install Alauda Container Platform Log Essentials via YAML

### 1. Check available versions

Ensure the plugin has been published by checking for ModulePlugin and ModuleConfig resources, in the `global`cluster:

```shell

# kubectl get moduleplugin | grep log-api
log-api                       30h
# kubectl get moduleconfig | grep log-api
log-api-v4.2.0                30h

```

This indicates that the ModulePlugin `log-api` exists in the cluster and version `v4.2.0` is published.

### 2. Create a ModuleInfo

Create a ModuleInfo resource to install the plugin without any configuration parameters:

```yaml
apiVersion: cluster.alauda.io/v1alpha1
kind: ModuleInfo
metadata:
  annotations:
    cpaas.io/display-name: log-api
    cpaas.io/module-name: '{"en": "Alauda Container Platform Log Essentials", "zh":
      "Alauda Container Platform Log Essentials"}'
  labels:
    cpaas.io/cluster-name: global
    cpaas.io/module-name: log-api
    cpaas.io/module-type: plugin
    cpaas.io/product: Platform-Center
  name: global-b4a67e6adb67bad47544354e9bf2bdff
spec:
  version: v4.2.0
  valuesOverride:
    ait/chart-alauda-log-api:
      global:
        nodeSelector:
          node-role.kubernetes.io/infra: "true"
        tolerations:
        - effect: NoSchedule
          key: node-role.kubernetes.io/infra
          value: reserved
          operator: Equal
```

YAML field reference:

| Field path | Description |
| --- | --- |
| `metadata.labels.cpaas.io/cluster-name` | Target cluster name where the plugin is installed. |
| `metadata.name` | Temporary ModuleInfo name; the platform will rename it after creation. |
| `spec.version` | Specifies the plugin version to install, must match `.spec.version` in ModuleConfig.|
| `spec.valuesOverride.ait/chart-alauda-log-api.global.nodeSelector` | Optional. The nodeselector for the Logging component. Use this parameter to select nodes for the logging component. This parameter is mainly used for infrastructure nodes. |
| `spec.valuesOverride.ait/chart-alauda-log-api.global.tolerations` | Optional. The tolerations for the Logging component. If the node selector selects tainted nodes, use this parameter to specify a taint toleration key, value, and effect. This parameter is mainly used for infrastructure nodes. |

### 3. Verify installation

Since the ModuleInfo name changes upon creation, locate the resource via label to check the plugin status and version:

```shell
kubectl get moduleinfo -l cpaas.io/module-name=log-api
NAME                                             CLUSTER         MODULE      DISPLAY_NAME   STATUS    TARGET_VERSION   CURRENT_VERSION   NEW_VERSION
global-e671599464a5b1717732c5ba36079795          global          log-api   log-api      Running   v4.2.0          v4.2.0           v4.2.0
```

Field explanations:

* `NAME`: ModuleInfo resource name
* `CLUSTER`: Cluster where the plugin is installed
* `MODULE`: Plugin name
* `DISPLAY_NAME`: Display name of the plugin
* `STATUS`: Installation status; `Running` means successfully installed and running
* `TARGET_VERSION`: Intended installation version
* `CURRENT_VERSION`: Version before installation
* `NEW_VERSION`: Latest available version for installation

## Install Alauda Container Platform Log Storage for Elasticsearch via console

1. Navigate to **Marketplace** > **Cluster Plugins** and select the target cluster.

2. Click the action button to the right of **Alauda Container Platform Log Storage for Elasticsearch** and select **Install**.

3. Refer to the following instructions to configure relevant parameters.

   | Parameter | Description |
   | --- | --- |
   | **External Elasticsearch** | Keep this switch off to install the platform-managed Elasticsearch storage plugin. |
   | **size** | Choose `Single Node`, `Small Cluster`, or `Big Cluster` based on the target scale. |
   | **Storage Method** | Choose `LocalVolume` or `StorageClass` for the storage backend. |
   | **Log Node** | Displayed when `LocalVolume` is used. Select the Elasticsearch data nodes. |
   | **Storage path on host** | Displayed when `LocalVolume` is used. Enter the absolute host path for Elasticsearch data. |
   | **Capacity** | Reserve storage capacity in Gi for the current data section. The page uses this label for both Elasticsearch data and Kafka data sections. |
   | **Log data nodes resources** | Configure CPU and memory requests and limits for Elasticsearch data nodes. |
   | **Kafka Nodes** | Select the Kafka nodes used by the plugin. |
   | **LogPlatform / LogWorkload / LogSystem / LogKubernetes / Kubernetes Event / Audit** | Configure retention days by log type in the **Log reserve config, in days** section. |

4. Click **Install**.

## Install Alauda Container Platform Log Storage for ClickHouse via console

### Prepare an S3 Secret

If you want to use S3 as the ClickHouse log data storage or cold data storage, first create an S3 Secret in the `cpaas-system` namespace of the target storage cluster.

The Secret must include the label `cpaas.io/credentials: S3`. The plugin installation page filters available Secrets by this label.

Example:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: clickhouse-s3-config
  namespace: cpaas-system
  labels:
    cpaas.io/credentials: S3
type: Opaque
stringData:
  ACCESS_KEY_ID: <access-key-id>
  SECRET_ACCESS_KEY: <secret-access-key>
  ENDPOINT: https://<s3-endpoint>
  REGION: <region>
  BUCKET: <bucket-name>
```

Field notes:

| Field | Description |
| --- | --- |
| `ACCESS_KEY_ID` | Access key for the S3 service. |
| `SECRET_ACCESS_KEY` | Secret key for the S3 service. |
| `ENDPOINT` | S3 service endpoint. |
| `REGION` | Region of the S3 service. Leave empty only if the backend does not require it. |
| `BUCKET` | Bucket used by ClickHouse. |

If the S3 endpoint uses `https`, also create the following CA certificate Secret in the same cluster before installing the plugin:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: clickhouse-ca-certificates
  namespace: cpaas-system
type: Opaque
stringData:
  ca-certificates.crt: |
    -----BEGIN CERTIFICATE-----
    <ca-certificate-content>
    -----END CERTIFICATE-----
```

HTTPS S3 CA Secret requirements:

| Field | Requirement |
| --- | --- |
| `metadata.name` | Must be `clickhouse-ca-certificates`. |
| `metadata.namespace` | Must be `cpaas-system`. |
| `stringData.ca-certificates.crt` | Replace only this field with the CA certificate chain required by the HTTPS S3 endpoint. |

1. Navigate to **Marketplace** > **Cluster Plugins** and select the target cluster.

2. Click the action button to the right of **Alauda Container Platform Log Storage for ClickHouse** and select **Install**.

3. Refer to the following instructions to configure relevant parameters.

   | Parameter | Description |
   | --- | --- |
   | **Size** | Choose `Single Node`, `Small Cluster`, or `Big Cluster` based on the target scale. |
   | **Storage Policy** | Controls how ClickHouse stores metadata and log data. Available values are **Default** and **Separation of storage and compute**. This setting cannot be changed after deployment. |
   | **Storage Method** | Displayed when **Storage Policy** is **Default**. Available values are `LocalVolume` and `StorageClass`. |
   | **Metadata Storage Type** | Displayed when **Storage Policy** is **Separation of storage and compute**. Available values are `LocalVolume` and `StorageClass`. |
   | **Log Node** | Displayed when local storage is used. Under **Default**, select the nodes that store ClickHouse log data locally. Under **Separation of storage and compute**, select the nodes that store ClickHouse metadata locally. |
   | **Storage Path on Host** | Displayed when **Storage Policy** is **Default** and **Storage Method** is `LocalVolume`. Default value: `/cpaas/data/clickhouse`. |
   | **Metadata Storage Path** | Displayed when **Metadata Storage Type** is `LocalVolume`. Default value: `/cpaas/data/clickhouse`. |
   | **Log Replicas** | Displayed when the selected storage mode uses `StorageClass`. Configure the number of ClickHouse replicas created for the active storage section. |
   | **StorageClass** | Displayed when **Storage Policy** is **Default** and **Storage Method** is `StorageClass`. Select the StorageClass used for ClickHouse log data. |
   | **Metadata StorageClass** | Displayed when **Metadata Storage Type** is `StorageClass`. Select the StorageClass used for ClickHouse metadata. |
   | **Capacity** | Displayed when **Storage Policy** is **Default**. Reserve capacity in Gi for ClickHouse log data. |
   | **Metadata Capacity** | Reserve capacity in Gi for ClickHouse metadata. The page recommends `20 Gi`. |
   | **S3 for Log Data** | Displayed when **Storage Policy** is **Separation of storage and compute**. Select a Secret in `cpaas-system` with label `cpaas.io/credentials: S3`. This setting cannot be changed after deployment. |
   | **Log Platform / Log Workload / Log System / Log Kubernetes / Kubernetes Event / Audit** | Configure hot-data retention days by log type in the **Log Retention Settings** section. |
   | **Cold and Hot Separate** | Enables tiered storage for ClickHouse. After it is enabled, you can configure an S3 location for cold data and set cold-data retention days by log type. This switch cannot be changed after deployment. |
   | **S3 for Log Cold Data** | Displayed when **Cold and Hot Separate** is enabled. Select a Secret in `cpaas-system` with label `cpaas.io/credentials: S3` for cold data storage. This setting cannot be changed after deployment. |
| **Log Platform / Log Workload / Log System / Log Kubernetes / Kubernetes Event / Audit** | When **Cold and Hot Separate** is enabled, the cold-storage settings section uses these same field names for cold-data retention. A value of `0` means that the log type is not transferred to cold storage. |

4. Click **Install**.

Upgrade notes:

- Upgrading the ClickHouse storage plugin does not allow you to switch the **Storage Policy** type or toggle **Cold and Hot Separate** on or off.
- After deployment, only the effective retention settings, such as hot or cold retention days, should be adjusted.
- To use this capability, upgrade **Alauda Container Platform Log Essentials** and **Alauda Container Platform Log Storage for ClickHouse** to compatible versions together.
- In disaster recovery scenarios, use different S3 buckets for the primary and standby ClickHouse deployments.

Field explanations:

* `NAME`: ModuleInfo resource name
* `CLUSTER`: Cluster where the plugin is installed
* `MODULE`: Plugin name
* `DISPLAY_NAME`: Display name of the plugin
* `STATUS`: Installation status; `Running` means successfully installed and running
* `TARGET_VERSION`: Intended installation version
* `CURRENT_VERSION`: Version before installation
* `NEW_VERSION`: Latest available version for installation

## Install Alauda Container Platform Log Collector Plugin via YAML

### 1. Check available versions

Ensure the plugin has been published by checking for ModulePlugin and ModuleConfig resources, in the `global`cluster:

```shell

# kubectl get moduleplugin | grep logagent
logagent                       30h
# kubectl get moduleconfig | grep logagent
logagent-v4.1.0                30h

```

This indicates that the ModulePlugin `logagent` exists in the cluster and version `v4.1.0` is published.

### 2. Create a ModuleInfo

Create a ModuleInfo resource to install the plugin without any configuration parameters:

```yaml
apiVersion: cluster.alauda.io/v1alpha1
kind: ModuleInfo
metadata:
  annotations:
    cpaas.io/display-name: logagent
    cpaas.io/module-name: '{"en": "Alauda Container Platform Log Collector", "zh": "Alauda Container Platform Log Collector"}'
  labels:
    cpaas.io/cluster-name: go
    cpaas.io/module-name: logagent
    cpaas.io/module-type: plugin
    cpaas.io/product: Platform-Center
    logcenter.plugins.cpaas.io/cluster: go
  name: <cluster>-log-agent
spec:
  config:
    crossClusterDependency:
      logcenter: go
      logclickhouse: null
    dataSource:
      audit: true
      event: true
      kubernetes: true
      platform: false
      system: false
      workload: true
    storage:
      type: ElasticSearch
  version: v4.1.0
  valuesOverride:
    ait/chart-alauda-log-agent:
      global:
        nodeSelector:
          node-role.kubernetes.io/infra: "true"
        tolerations:
        - effect: NoSchedule
          key: node-role.kubernetes.io/infra
          value: reserved
          operator: Equal
```

YAML field reference (Log Collector):

| Field path | Description |
| --- | --- |
| `metadata.annotations.cpaas.io/display-name` | Plugin display name. |
| `metadata.annotations.cpaas.io/module-name` | Plugin i18n name JSON string. |
| `metadata.labels.cpaas.io/cluster-name` | Target cluster where the plugin is installed. |
| `metadata.labels.cpaas.io/module-name` | Must be `logagent`. |
| `metadata.labels.cpaas.io/module-type` | Must be `plugin`. |
| `metadata.labels.cpaas.io/product` | Product identifier, typically `Platform-Center`. |
| `metadata.labels.logcenter.plugins.cpaas.io/cluster` | Storage cluster name to which logs are pushed when the collector uses the Elasticsearch storage plugin. |
| `metadata.labels.logclickhouse.plugins.cpaas.io/cluster` | Storage cluster name to which logs are pushed when the collector uses the ClickHouse storage plugin. |
| `metadata.name` | Temporary ModuleInfo name; the platform will rename it after creation. |
| `spec.config.crossClusterDependency.logcenter` | Name of the Elasticsearch-based log storage cluster. |
| `spec.config.crossClusterDependency.logclickhouse` | Set to `null` when using Elasticsearch storage; otherwise set to ClickHouse cluster name. |
| `spec.config.dataSource.audit` | Collect audit logs. |
| `spec.config.dataSource.event` | Collect event logs. |
| `spec.config.dataSource.kubernetes` | Collect Kubernetes logs. |
